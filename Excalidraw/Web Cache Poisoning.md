When the web server returns data (Step 3)
it has to include a data (Cache Header) regarding whether
certain content can be cached or no

(Ex: /private-key, /settings)

You can't cache these paths because 
they are user dependent ^9daeEZCz

A Risk
The response that comes back from the web server contains a body. 

If you were to inject something to the body and it would get cached, 
then what would happen is that the next person who visits the website 
would get cached version of the website where you have written something

This would be web cache poisoning. This can be done independent of HTTP status code
Meaning that status codes such as 200 OK and 404 Not Found can both be poisoned ^og3HOpFm

How to understand if there is a cache implementation?

Make a request (with random query string) and look at the response time
If you make another request and this time it is quicker, it means there is caching

Another way of doing this is by looking at the HTTP response ^esfEl4ZP

How do you think web frameworks (Ex: Laravel) implement this
 ^BaHsoDDX

They look at request and build it up from there. 
(Host for url, HTTP /GET part for protocol)

Modifying response X-Forwarded-Host to a different domain 
would change these urls too ^GyAimRgn

Keyed and Unkeyed values

Certain values are keyed (like query strings) and some others aren't ^J9WeaE9f
